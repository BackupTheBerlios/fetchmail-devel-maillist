On Thu, Jul 21, 2005 at 04:44:49PM +0100, Rob MacGregor wrote:
> On 21/07/05, Brian Candler <B.Candler at pobox.com> wrote:
> > 
> > (Note: we're not talking about "users" here; we're only talking about people
> > who already have root access to their system. There are a million stupid
> > things they could do. Running fetchmail as root is pretty minor in
> > comparison to many of the others).
> 
> Running fetchmail in no way requires root access - any user account
> will do.  So, if you've shell access on a machine, you could slap the
> binary on and run it...

Errm, I think we're saying the same thing.

If you have a normal user account:
  => you can run fetchmail, and this is just fine and dandy

If you are logged in as root:
  => you can switch to a normal user instead and run fetchmail (recommended)
  => you can run fetchmail as root: stupid, but not as stupid as many other
     things you could do as root.

In the last case, all I'm saying is that if the system administrator is that
stupid, then making fetchmail *force* them to run as a non-root user is
pretty pointless, as they are almost certainly doing worse things anyway.

I can only see value if you want to engage in a blame-shifting exercise:
i.e. if a user runs fetchmail as root, AND a security hole in fetchmail
exists, AND it is exploited by a hostile pop3 or smtp server, THEN you can
say "don't blame us, we told you not do run it as root".

But you can achieve that just in the documentation, and indeed the GPL's "no
warranty" clause is very clear.

In any case - there are lots of Unix commands which don't *need* to be run
as root, and which could put you at risk if you do, but they still let you.
What about 'telnet' and 'ssh' for example?

Regards,

Brian.
