> On Wed, Jul 20, 2005 at 02:22:32PM +0200, Nico Golde wrote:
>> is there really a use for fetchmailconf? I mean are there
>> users who use it? Most people who use fetchmail are familar
>> with mta's procmail etc. and do they really need such a
>> script?
>> What about dropping it (don't flame me :)
>
> I think that firstly fetchmail's configuration file should be made much more
> logical and self-consistent.
>
> I've had difficultly wrapping my head around this before; if I
> understand correctly, there are:
> - global options
> - per-server options
> - per-account options
>
> but (a) these are not clearly demarked, and (b) there seems to be little
> logic as to which option is where. Yet try to use an option in the wrong
> place at it bombs out.

Well, the right place for the option depends on how it's implemented
internally. From a user POV, it's pretty nasty this way. I haven't had
the time to look if this can be simplified.

Your other suggestions about the inheritance scheme are sound, but a
mile or two away from the current implementation, and not a priority
item. There are some issues borne in the design that are way more
important, for instance making sure fetchmail runs without
privileges. After that we'll see if we can get closer to the ISC dhcpd
configuration scheme, which has this "inheritance" or however you call
it scheme.

> P.S. Perhaps Python-style indenting would be more readable?

Now that you mentioned it, it will have to support both. :)
