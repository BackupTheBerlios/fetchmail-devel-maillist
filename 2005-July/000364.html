Brian Candler wrote:
> I think the problem is not that the local smtp server is rejecting the
> mail with a 501 response (which it is entitled to do, for any policy
> reason); the problem is that the SMTP conversation is getting stuck at
> this point. Fetchmail ought to be able to recover from this situation.

Fetchmail's header validation has long been a source of stuck mail 
problems, but if I remember correctly, it's done so that fetchmail can 
bounce the mail if necessary, as well as not be responsible for passing 
bad headers downstream.

Part of the problem is that fetchmail seems to be operating in an 
SMTP-style mode where invalid headers can be rejected without first 
accepting the mail. POP and IMAP we really can't just refuse to accept a 
message with an invalid sender address and expect the upstream sender to 
fix it.  We need to take the mail and do *something* with it.

This is something I don't think we can fix for the upcoming release, but 
should be on the list of things to address in the following release.
